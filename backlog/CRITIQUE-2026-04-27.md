**Context.** Hareesh asked why WhatsApp self-chat messages weren't being processed by Gemini. Diagnosis pulled threads and uncovered five latent bugs in code shipped over the prior two weeks, plus a meta-pattern that all five share. This document captures them so they aren't silently absorbed; the fixes belong in upcoming sessions, sequenced per `docs/EXPERIENCE-AUDIT.md`.

**Hareesh's verbatim feedback at the end of the session:**

> "What I am quite annoyed to be honest about is the fact that when you're designing, and pushing through these codes, we're introducing new bugs or building stuff that's actually not efficient, etc. … I spent nearly 100k token so far just to uncover issues in the code that you built."

This isn't framed as Anthropic owing money back. It's framed as a coherent failure mode in how features have been shipped: each slice passes its own tests in isolation, nobody walks through the felt experience, and the gaps survive review after review. ## Bug 1 — WhatsApp self-chat dies in dead inbox queue

**Severity:** High. Blocks the only WhatsApp ingestion path the 2026-04-26 gate left open.

**What happens:** The `messages.upsert` handler in `src/channels/whatsapp.ts:111` correctly passes self-chat messages through `shouldIngest` (verified against ownJid `447462616146:17@s.whatsapp.net` after `:17` device-suffix stripping). The message reaches `handleIncomingMessage` in `src/intelligence/orchestrator.ts:277`. There, group messages (`@g.us`) fire `processGroupMessageExtraction` immediately. Self-chat and DMs fall through to `inbox_messages` for "Phase 1: Omnivorous Batched Ingestion" (line 335-338 comment).

The batch worker doesn't exist. `inbox_messages` accumulated zero rows in the last 24h on the live Z2 (verified by query) despite 60+ self-chat sends being skipped or queued. Self-chat → extraction was never wired.

**Fix:** One-line change at `orchestrator.ts:330`:

```ts
const ownJidNormalised = normaliseJid(sock.user?.id);
const senderNormalised = normaliseJid(senderJid);
if (senderJid.endsWith('@g.us') || senderNormalised === ownJidNormalised) {
  await processGroupMessageExtraction(profileId, content, senderJid, msg.key?.id || 'unknown');
  return;
}
```

Plus tests covering self-chat routing.

**Root cause:** Architecture had the *shape* of "we'll batch process" but the worker was never built. The 2026-04-26 self-chat gating change accidentally codified a no-op as the default behaviour.

## Bug 2 — `stream_cards.source` CHECK rejects mobile/pwa/whatsapp_dm channels

**Severity:** High. Silently drops every extracted card from chat surfaces. Never noticed because the failure is logged and discarded.

**What happens:** `extraction.ts:46` writes `source = senderJid` for non-group messages (everything except `@g.us`). The CHECK constraint at `schema.sql:294` only allows `'whatsapp_group', 'calendar', 'email', 'document', 'manual', 'proactive'`. So `source = 'mobile'`, `source = 'pwa'`, and `source = '447462616146@s.whatsapp.net'` all violate the CHECK and the INSERT fails.

Confirmed in live logs:
```
error: new row for relation "stream_cards" violates check constraint "stream_cards_source_check"
Failing row contains (... mobile, mobile-1777290769726-470biv, ...)
```

**Fix:**
1. Migration extending the CHECK to include `'whatsapp_dm', 'whatsapp_self', 'mobile', 'pwa'`.
2. Update `extraction.ts:46` to map channel → source via a dedicated function rather than passing through raw JID:
   ```ts
   function channelToSource(channel: string): string {
     if (channel.endsWith('@g.us')) return 'whatsapp_group';
     if (channel.includes('@s.whatsapp.net')) return 'whatsapp_dm';
     if (channel === 'mobile') return 'mobile';
     if (channel === 'pwa') return 'pwa';
     return 'manual';
   }
   ```
3. Tests covering each channel mapping.

**Root cause:** The schema enum was written for an architecture where extraction only fired from WhatsApp groups. When mobile and PWA chat paths started calling extraction, no one updated the schema.

## Bug 3 — Push tokens never registered for any user

**Severity:** Critical. The morning briefing — the primary "sticky" behaviour that gives users a daily reason to open Memu — has never been delivered as a push notification to any user, ever.

**What happens:** Empty `push_tokens` table verified by query against the live Z2:
```
 profile_id | display_name | token | created_at | last_seen_at
------------+--------------+-------+------------+--------------
(0 rows)
```

`pushMorningBriefingToMobile` in `briefing.ts:269` early-returns at line 274-276 when `getTokensForProfile` returns empty. Sonnet is never even called, no `[BRIEFING PUSH]` log line ever appears. This is silent — there's no error, no retry, no warning. The 22 ledger entries observed for the `briefing` skill on 2026-04-27 came from `/api/dashboard/synthesis` (the Today screen), not from the cron.

The mobile app's push registration code path runs at `_layout.tsx` boot. It requires `Device.isDevice` (simulators return null) AND `Notifications.requestPermissionsAsync` granting permission. If the user taps "Don't allow" once at first launch, registration silently fails. There's no in-app way to re-request permission or re-trigger registration.

**Fix:**
1. Investigate why registration is failing for Hareesh's actual phone (Device.isDevice should be true, permission likely granted given other apps).
2. Add a diagnostic Settings row: "Push notifications: <status>" with a retry button.
3. Log push token registration attempts and failures so future debugging has a trail.
4. After registration succeeds, send a one-off "Push notifications enabled" silent confirmation push so the user knows it's working.

**Root cause:** The registration code was written but never tested end-to-end on a real device with the full lifecycle. It works "in theory" — passes its slice tests — but no one ever confirmed a token landed in the database from a real user's first install.

## Bug 4 — Today screen re-fires Sonnet on every render (no caching)

**Severity:** Medium-high. ~£0.40-0.50/day quietly burned at single-user volume; multiplies linearly with users; will exceed the £2.99 monthly subscription per user at 10+ tab-switches/day.

**What happens:** `loadSynthesis()` in the mobile Today screen calls `/api/dashboard/synthesis` inside `useFocusEffect` with no cache, no debounce, no data-availability gate. Every focus event (tab switch, screen unlock with Memu open, returning from another screen) fires a fresh Sonnet call via `generateProactiveSynthesis` → `runUnifiedBriefing`.

Observed pattern in the privacy_ledger on 2026-04-27 for Hareesh's profile alone:
- 02:26-29 (3 calls in 3 min) — middle of night, likely background screen wake
- 06:17-21 (7 calls in 4 min) — pre-dawn check + tab navigation
- 12:04-11, 12:52, 15:02 — daytime check-ins

Each call ~£0.018. 21 calls in one day for one user = £0.38. At 20 Founding-50 families × 2 active adults each, this scales to ~£15/day in cost-on-render before any user does anything intentional. That's ~£450/month in waste for a £60/month revenue line.

**Worse:** when context is empty (most of Day 1-7), Sonnet is generating "you have a quiet day, enjoy" platitudes from nothing. The cost is buying confabulation.

**Fix:**
1. Cache synthesis result for N minutes (15-30) keyed on `(profileId, date)`. Cache busts on new stream card, new calendar event, new explicit user prompt.
2. Gate behind data-availability check: if no calendar, no stream cards, no context entries → return a static onboarding prompt, don't call Sonnet.
3. Replace `useFocusEffect` with explicit refresh triggers (pull-to-refresh, app foreground after >1h).

**Root cause:** The synthesis card was specced as "always fresh on Today" without anyone calculating the runtime cost. Classic cost-on-render anti-pattern that's invisible until the bill arrives.

## Bug 5 — `generateAndPushMorningBriefing` is misnamed and doesn't push; PWA header says "Morning briefing / Good evening"

**Severity:** Medium. Function name lies about what it does; PWA renders incoherent two-string header to the user.

**What happens:**

(a) `generateAndPushMorningBriefing` in `briefing.ts:262` calls `runUnifiedBriefing(profileId, 'app')` and returns the result. **It does not push anywhere.** The name is from an earlier design where this was the WhatsApp delivery path, never updated when the WhatsApp delivery was retired. The cron at `index.ts:2174` calls it for every WhatsApp-channel profile and discards the result.

(b) The PWA Today screen renders the literal string "Morning briefing" as the section header at all times of day, while the body of the briefing is generated by Sonnet which adopts time-of-day greetings. Result observed by Hareesh on the live PWA at evening:
```
Morning briefing
Good evening, Hareesh
Monday 27 April.
```

The header is hardcoded; the body is dynamic; nobody designed the relationship between them.

**Fix:**
1. Rename `generateAndPushMorningBriefing` → `generateBriefingText` (it generates only). Or delete and replace with `runUnifiedBriefing` directly — it's a one-line wrapper.
2. PWA header should match body register. Either: (a) make the header dynamic ("Today's brief" / "Where you are right now" — time-neutral), or (b) constrain the briefing skill to a fixed header-matching greeting per time window.
3. Same fix on the mobile Today screen if the same pattern exists there.

**Root cause:** Two code paths writing two strings to one user-visible composition without ever being looked at together. Classic walk-through-omission.

## Bug 6 — git ENOENT in Docker image breaks every clean install

**Severity:** Critical for any new family / clean Z2 install. Documented in `INBOX.md` 2026-04-26 as part of the autolearn ship; surfaced again by the conviction-and-specialist cluster audit.

**What happens:** `spaces/store.ts` `ensureFamilyRepo()` calls `git init` un-wrapped. The rebuilt API container is missing the git binary on PATH. PDF upload exposed it 2026-04-26 (every Space write that needs to init a new family repo crashes with `spawnSync git ENOENT`). Hareesh's existing family is unaffected because `.git` already exists from previous Space writes; the inner `git add`/`git commit` ops are try/catch wrapped — they warn but don't fail.

For any new family / clean Z2 setup / first Founding-50 invitee, this is blocking.

**Fix:**
1. Install git in the Dockerfile (`apt-get install -y git` in the base layer).
2. Defensively wrap `ensureFamilyRepo` so a missing git binary degrades to filesystem-only operation rather than crashing the request. Log a warning with concrete remediation guidance.
3. Smoke test: spin up a fresh container against an empty database and confirm the first Space write succeeds.

**Root cause:** Container was rebuilt without git on PATH (likely a base image change or Dockerfile drift), and the code that uses git assumes it exists.

## Meta-bug — AI confabulates from emptiness instead of admitting the gap

**Severity:** Critical for trust. Cross-cutting across all five concrete bugs above and across 16 personas in the experience audit.

**What happens:** When Memu has no/sparse data — empty calendar, empty stream cards, empty context_entries — it does not say so. `runUnifiedBriefing` passes "No new messages." and "No events in the next 48h." to the briefing skill. The skill is instructed to set `has_substantive_updates: false` in that case. But:

- `generateProactiveSynthesis` (no `has_substantive_updates` gate) runs Sonnet anyway and may emit "you have a quiet day, enjoy the calm" which sounds like confident insight.
- The chat path (`interactive_query`) takes any retrieval result — including empty — and produces a confident answer from training data. There's no "I have no notes on that yet" fallback. The Privacy Ledger shows what was sent but does not flag "context block was empty."
- The LLM's tone (warm, conversational, knowing) is calibrated for a system that knows the user. When applied to empty context, it sounds like assertive guessing dressed as memory.

The four cluster audits flag this independently:
- **Household core:** "the briefing might say 'you have a super quiet day, enjoy' from empty data" → admin churn signal.
- **Adult roles:** "for medical information, unsourced AI output is actively unsafe" → caregiver of aging parent has a safety risk.
- **Career and growth:** "for career-development queries, a hallucinated answer is worse than no answer because the user may act on it" → trust destroyer.
- **Conviction and specialist:** "a journalist asking 'what do I know about [investigation]' on empty database gets a confident AI response synthesised from general knowledge, not from their notes" → highest-risk single failure mode in the entire audit.

**Fix:** Cross-cutting change, not a single line:

1. **Empty-state gate on synthesis paths.** Before calling Sonnet, check: any calendar events? any stream cards? any context_entries? If all empty, return a structured "no data yet — here's what to seed" response. No LLM call.
2. **Retrieval-empty flag on chat replies.** When `interactive_query` retrieves zero relevant Spaces and zero context_entries, the response should be prefixed with a marker the UI can render distinctly — "I don't have anything in my notes about this; here's what I can offer from general knowledge instead." Never blend "from your notes" and "from training data" without the line between them being legible.
3. **Source citation in chat UI.** The provenance infrastructure exists in `src/spaces/provenance.ts`. Pass it through the API response and render it under retrieval-grounded replies. This is the affirmative side of the gate — when retrieval *did* find something, say so.
4. **Privacy Ledger flag.** Extend the ledger schema with `context_block_empty: boolean` so a user auditing it can see at a glance which of their interactions were grounded.

**Root cause:** The product was built assuming context would accumulate before users started asking questions. In practice — for every persona — Day 0 Memu is asked questions before context exists. The product has no honest empty-state behaviour, only a confident generative one. This is the single highest-leverage change across the entire product surface.

## Sequencing the fixes

These six bugs should not all land in one session. The experience audit (`docs/EXPERIENCE-AUDIT.md`) sequences them across 4-5 build sessions interleaved with the audit's positive recommendations (onboarding seed, source citation, memory visibility, domain scoping). The order below reflects bug-only sequencing within those sessions:

| Session | Fixes |
|---|---|
| Foundation (next session) | Bug 6 (git ENOENT) → Bug 3 (push tokens diagnosis + retry path) → Bug 4 (synthesis cache + empty-state gate) |
| Foundation 2 | Bug 1 (self-chat → extraction routing) → Bug 2 (stream_cards CHECK + channel mapping) → Bug 5a (rename misnamed function, delete unused WhatsApp branch) |
| Onboarding | Bug 5b (header/body coherence as part of broader empty-state work) → Meta-bug part 1 (empty-state gate on synthesis) |
| Memory & provenance | Meta-bug parts 2-4 (chat retrieval-empty flag, source citation in UI, ledger flag) |

**The meta-bug isn't fixed by patching code paths individually — it's fixed by treating "what does Memu say when it doesn't know" as a first-class product capability.** That is the heart of why this audit exists.
